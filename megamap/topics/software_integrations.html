Software Integrations

Background

Quuppa Software Suite provides you with a full set of tools for getting efficiently started with setting up your (first) RTLS application or project.

With Site Planner, you can draw and configure your project - setting up the floorplans, geofences, tag configurations, data formats etc.

System Simulator is integrated within the Site Planner, and with the Simulator, you can push simulated use cases (simulation paths) to your Positioning Engine. This way, you can start integrating your use cases into your own application(s) before setting up any hardware.

The simulator will also provide you with a recommendation for QPE hardware specifications. Factors that affect the hardware requirements depend on the things such as desired level of accuracy (number and type of Locators) and the scale (size of the area and number of tracked assets).

Positioning Engine

In order to run a simulation, a Positioning Engine is needed to process the simulated data. The Quuppa System Simulator is a realistic simulation tool that produces raw data similar to the real environment, and the Positioning Engine processes the data.

Setting up the Positioning Engine for your simulation is included in the Quuppa Customized Simulation package. Depending on your needs and preferences, Quuppa can set up the Positioning Engine in a hardware server (QPE Classic) or in a cloud instance (QPE Enterprise).

If you have not purchased the Quuppa Customised Simulation package and want to set up the Positioning yourself, follow the instructions on Quuppa Online Documentation to set up your engine.

Setting up a site plan

The site layout and simulated use cases are created using the Site Planner tool. It’s good to remember that if you are preparing for a real-world installation, it is recommended that while building your simulation, you should make it as realistic as possible (scaling of the background, placing the zones and openings, etc.). This way, you could re-use the same site plan and project for your future real world deployment.

  1. Start creating the site plan by selecting ‘New’ in the ‘Project’ menu.
  2. Select basic settings for your project (Operating channels, Locator types and default installation heights, etc.)
  3. Import and scale the background image for your project
  4. Frame the area that you are doing tracking in (Tracking Area)
  5. Add Locators manually or using the ‘Locator distributor’ feature
  6. Check that the coverage is good using the ‘Render coverage’ feature
  7. Specify physical areas such as rooms by creating ‘Smart Zones’ (Smart Edge)
  8. Specify geofences by creating ‘Smart Zones’ (Hidden Areas and Location based commands)
  9. Specify semantic areas by creating Zones (Zones without any actions)
  10. Create tag groups for all the different types of assets you are tracking (especially if they are tracked with different settings) (Not in the video)

Creating a simulation

Once your site plan is created (or you received a ready-made site plan), you can start adding simulation paths and use cases to your plan. In the simulation, for the tags that move on the path, you can specify:

  • Name prefix
  • Number of tags
  • Speed of tags
  • Tag group - which determines what type of settings the tags are tracked with (transmit rate, etc.)
  • Stopping points
    • For each point on the path you can set the tag to stop for specified number of seconds

Example 2: Customising the API response for your own application

When polling data from the Quuppa System through the API there are 4 pre-defined data formats available:

  • defaultInfo
    • Provides basic information about the tag status, such as name, last seen, battery voltage etc.
  • defaultLocation
    • Provides basic information about the tags’ location, such as (carthesian) coordinates (x, y, z), movement status (moving or stationary), zone, etc.
  • defaultLocationAndInfo
    • Provides both tag status information and location information.
  • ALL_ITEMS
    • Provides all available data fields

When using the /getTagData API method, you can specify which format to use by using the ‘format’ parameter (for example format=defaultLocation)

It is also possible to customise the format when requesting data. This can be done using the ‘formatDefinition’ parameter. In the ‘formatDefinition’ parameter you can supply a list of fields you want to receive in Quuppa API syntax, separated by comma.

Pushing data using the QPE Output Targets

Creating / modifying output format to optimise data

API formats are used to specify which data points from a tag you want to receive. Depending on your application or use case, you might not need all of the available data points at all times and in order to avoid unnecessary data traffic you can create and use different output formats to reduce the amount of data you receive.

In the default settings of a Site Plan, there is a set of pre-defined API (data) formats for you to get started with. These formats are:

  • defaultInfo
    • Provides basic information about the tag status, such as name, last seen, battery voltage etc.
  • defaultLocation
    • Provides basic information about the tags’ location, such as (carthesian) coordinates (x, y, z), movement status (moving or stationary), zone, etc.
  • defaultLocationAndInfo
    • Provides both tag status information and location information.

In the example implementation we will create a new output format that will also include the geo-coordinates (latitude and longitude) for a tag.

Creating an output target and sending a data stream

The data from Quuppa Positioning Engine can be received either by polling information from the API (ie. /getTagData) OR by pushing data out from the engine.

The outgoing push streams are configured & controlled by the ‘Output Targets’ in your site plan.
